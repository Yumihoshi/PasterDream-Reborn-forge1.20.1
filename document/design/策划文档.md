# 策划文档

> PasterDream Reborn —— 源模组重置项目策划

---

## 一、项目背景

**PasterDream Reborn** 是对旧版 PasterDream 模组的重置项目。源模组在早期开发中存在诸多不规范问题，本次重置的目标是在保留核心玩法与美术风格的前提下，对代码架构、注册体系、命名规范进行全面整改。

---

## 二、源模组不规范问题总览(待补充)

| 类别 | 问题描述                                            | 重置对策 |
|------|-------------------------------------------------|---------|
| **注册 ID** | 部分方块/物品 ID 无意义（如 `grass_` + 数字后缀）或不符合作者命名习惯     | 统一重新命名，建立旧→新 ID 映射表 |
| **客制化类** | 部分方块/物品使用了不必要的客制化类（仅为了满足简单功能而创建独立类）             | 评估必要性：无特殊机制的方块/物品改用原版或通用类 |
| **配方名称** | 配方文件名无意义（`crafting_0.json`、`crafting_1.json` 等） | 见 [命名规范文档](./命名规范文档.md) |
| **材质命名** | 大部分材质文件使用中文拼音命名，与注册 ID 不匹配                      | 见 [命名规范文档](./命名规范文档.md) |
| **MapColor** | 源模组大多数方块未设置 `mapColor`，导致地图上显示颜色不正确               | 注册方块时**必须**设置 `.mapColor()` |
| **方块标签** | 源模组大量方块缺少挖掘工具标签（`MINEABLE_WITH_*`）、材质标签等，导致挖掘速度异常或与其他模组不兼容 | 搬运时**必须**补全标签：挖掘工具、材质、功能标签 |
| **包结构** | 源模组包结构扁平化，所有类集中在少数包中                            | 按功能模块分包（详见 [架构规范](../rule/program/架构规范.md)） |

## 三、ID 映射表规范

### 3.1 映射表格式

每次搬运方块/物品时，无论是否修改 ID，都必须在 ID 映射表中登记。映射表文件见 [ID映射表](./ID映射表.md)。

```markdown
| 类型 | 旧 ID | 新 ID | 变更说明 |
|------|-------|-------|---------|
| 方块 | `dyedream_dirt` | `dyedream_dirt` | 无需改动，直接搬运 |
| 方块 | `qym_doll_0` | `qym_doll` | 去掉数字后缀，统一为模型变体 |
| 物品 | `crafting_42` | `dream_shard` | 重命名，有意义的英文名 |
```

### 3.2 ID 命名规则

- **全部小写**，单词用下划线分隔（snake_case）
- **语义化**：名称应反映物品/方块的用途或外观
- **禁止**：中文拼音、无意义数字编号、过长的名称（建议不超过 4 个单词）
- **变体处理**：同系列用后缀区分，如 `cyan_stone_brick_slab`、`cyan_stone_pillar`

## 四、客制化类的处理原则

### 4.1 评估标准

在搬运方块/物品时，判断源模组中是否有不必要的客制化类：

| 是否需要客制化类 | 判定条件 | 示例 |
|----------------|---------|------|
| ❌ **不需要** | 方块仅使用原版属性（硬度、爆炸抗性、音效），无特殊交互逻辑 | 普通建筑方块：`CyanStoneBlock` |
| ❌ **不需要** | 物品没有任何右键行为、属性变更、或 tick 逻辑 | 普通素材物品：`AmberCandy` |
| ✅ **需要** | 方块有特殊交互（右键打开 GUI、红石信号响应、实体碰撞等） | `DreamCauldronBlock` → `DreamCauldronBlockEntity` |
| ✅ **需要** | 物品有右键行为、耐久系统、属性修饰、tick 逻辑等 | `SquealWaveWandItem` |
| ✅ **需要** | 方块/物品需要自定义模型渲染或动画 | `WindKnightSpawnblock` |

### 4.2 处理流程

1. 检查源模组中方块/物品的实现类
2. 按上表评估是否必要
3. **不需要的情况**：在 `ModBlocks`/`ModItems` 中直接用 `new Block(Properties...)` / `new Item(Properties...)` 注册
4. **需要的情况**：保留客制化类，但进行代码规范化（命名、结构、注释）

## 六、工作流规范

### 6.1 搬运前的准备

1. **明确搬运目标**：列出本次要搬运的方块/物品清单
2. **源项目定位**：在 `reference/` 中找到对应的源代码
3. **规范性分析**：检查以下各点并记录：
   - ID 是否规范？→ 记入 ID 映射表
   - 是否使用了不必要的客制化类？→ 记录处理方式
   - 材质命名是否规范？→ 如需重命名，记录材质映射
   - 配方（如有）命名是否规范？→ 记录配方重命名方案
   - 标签是否完整？→ 记录缺失的挖掘工具、材质、功能标签
4. **编写搬运方案**：简述如何改、如何重写

### 6.2 方块搬运清单流程

以搬运一个方块为例的标准流程：

1. **[ID 映射]** 无论是否改 ID，登记到 [ID映射表](./ID映射表.md)；登记后运行 `python generate_id_mapping.py` 刷新 `mapping.json`
2. **[注册]** 在 `ModBlocks.java` 中注册：
   - 无特殊机制 → 直接用 `new Block(Properties...)`，**必须**设置 `.mapColor()`
   - 有特殊机制 → 创建客制化类，放在 `world/level/block/custom/` 包下（如不存在则创建），类内设置 `mapColor`
3. **[物品关联]** 在 `ModItems.java` 中注册对应的 `BlockItem`
4. **[数据生成]** 在以下 Provider 中注册：
   - `ModBlockStateProvider` → 方块状态 + 模型
   - `ModItemModelsProvider` → 物品模型（block item）
   - `ModBlockLootTablesProvider` → 战利品表
   - `ModBlockTagsProvider` → 方块标签（**必须**检查并补全：挖掘工具、材质类型、功能标签）
5. **[语言文件]** 在 `ModEnUsLangProvider` 和 `ModZhCnLangProvider` 中添加
6. **[材质]** 添加纹理文件到 `assets/pasterdream/textures/block/`（按命名规范命名）
7. **[创造模式物品栏]** 在 `ModCreativeModeTabs` 中添加
8. **[配方搬运]** 在 `document/reference/old-recipes/` 中按旧 ID 搜索配方文件，确认后写入 `ModRecipesProvider`（如需切石机配方，注意产物与输入的关系）
9. **[数据生成运行]** `runData` 生成所有数据文件（blockstate、model、loot_table、lang、tags、recipes）
10. **[客户端测试]** `runClient` 验证：模型显示、纹理正确、中英文名称、合成/烧制配方

### 6.3 物品搬运清单流程

1. **[ID 映射]** 登记到 ID 映射表
2. **[注册]** 在 `ModItems.java` 中注册：
   - 无特殊机制 → 直接用 `new Item(Properties...)`
   - 有特殊机制 → 创建客制化类，放在 `world/item/custom/` 包下（如不存在则创建）
3. **[数据生成]** 在以下 Provider 中注册：
   - `ModItemModelsProvider` → 物品模型
   - `ModItemTagsProvider` → 物品标签（**必须**检查并补全）
4. **[语言文件]** 添加中英文
5. **[材质]** 添加纹理（如需要）
6. **[创造模式物品栏]** 添加到物品栏
7. **[配方搬运]** 在 `document/reference/old-recipes/` 中按旧 ID 搜索配方文件，确认后写入 `ModRecipesProvider`
8. **[数据生成运行]** `runData` 生成数据文件（model、lang、tags、recipes）
9. **[客户端测试]** `runClient` 验证
