# 程序规范

## Git 管理

### 仓库

- **单加载器 + 单 Minecraft 版本**：本仓库仅对应 Forge 加载器与 Minecraft 1.20.1，不涉及多加载器或多版本兼容
- **可见性**：在首个正式版发布之前，代码仓库始终保持私有状态

### 分支

- **主分支**：`master` 分支，始终保持与最新开发进度同步
- **开发者个人分支**：命名为 `dev-<开发者姓名>`（如 `dev-shiliuyinyu`）

### 提交

**Commit message 格式**：`[type](scope): message`

| type     | 说明                         |
| -------- | ---------------------------- |
| feat     | 新增玩法/功能                |
| fix      | 修复逻辑/表现 bug            |
| docs     | 仅文档变更                   |
| style    | 代码样式调整（无运行影响）   |
| refactor | 重构/代码清理                |
| perf     | 性能优化                     |
| test     | 添加或修改测试               |
| chore    | 更新工具/辅助脚本/杂项       |
| art      | 替换/更新美术资源            |
| asset    | 更新数据表/配置              |

**示例**：

- `feat(block): 添加梦之釜多方块结构`
- `fix(entity): 修复梦魇实体 AI 目标选择异常 (Fixes #42)`
- `refactor(network): 重构数据包同步逻辑`
- `docs(readme): 更新安装说明`
- `chore(build): 添加 Curios API 依赖`

> 单个 Commit message 可以带多个 `[type](scope): message` 格式的消息。

## 代码风格 & 命名规范

- **命名体系**：严格遵循 Java 驼峰命名规范
- **类名**：大驼峰（PascalCase），如：`DreamCauldronBlock`
- **方法名**：小驼峰（camelCase），如：`getDreamEnergy()`
- **常量**：全大写 + 下划线分隔，如：`MAX_DREAM_ENERGY`
- **特殊规则**：
    - **复数限制**：仅允许用于集合/列表类变量（如 `items`、`blockTags`），普通变量或类名禁止使用复数
    - **数据生成器后缀**：所有数据生成类须以 `Provider` 结尾，如 `ModBlockLootTablesProvider`
    - **注册中心类前缀**：以 `Mod` 为前缀，如 `ModItems`、`ModBlocks`
    - **方块实体类后缀**：以 `BlockEntity` 为后缀，如 `DreamCauldronBlockEntity`

## 易犯错误记录

### 官方映射下前置模组 Mixin 兼容问题

当使用 `official`（Mojang）映射渠道时，部分第三方模组（如 Curios、GeckoLib）的 `refmap.json` 中 `mappings` 段可能缺失或使用了 SRG 名称而非 Mojang 名称，导致开发环境（`runClient`/`runData`）中 Mixin 注入失败。

**症状**：`InvalidAccessorException: No candidates were found matching f_xxx_` 或 `InvalidInjectionException: @Inject annotation could not find any targets matching 'm_xxx_'`

**解决方案**：修补第三方 jar 中的 refmap，将 `mappings` 段的条目修正为 Mojang 名称。生产环境不受影响（混淆后使用 `searge` 段的 SRG 名）。

### 空 Mixin 配置文件

即使模组自身不使用 Mixin，当前置模组（如 Curios、GeckoLib）触发了 Mixin 加载时，模组也需要提供一个空的 `{modid}.mixins.json`，并在 jar manifest 中声明 `MixinConfigs`。

### GeckoLib 示例物品栏

GeckoLib 在开发环境中会注册 "Geckolib Examples" 创造模式物品栏。生产环境自动关闭。开发环境可通过 JVM 属性 `geckolib.disable_examples=true` 关闭，已在 `build.gradle` 的 `configureEach` 中配置。

## 审核流程

1. **提交前自查**：本地编译通过，无语法与逻辑错误。推送前先拉取最新更改
2. **合并请求流程**：向 `master` 或版本分支发起合并请求时，必须关联相应 Issue。必须填写清晰的变更描述

## AI 规范

AI 工具可以用于辅助开发，但需注意：

- AI 生成的代码必须经过人工审查，确保逻辑正确、风格一致
- 不得将敏感信息（密钥、令牌等）输入 AI 工具
- AI 辅助生成的代码如涉及第三方版权代码，需标注来源
