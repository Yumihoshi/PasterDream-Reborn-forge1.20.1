## 四、架构原则

1. **客制化类最小化** — 无特殊机制直接用原版类，仅特殊逻辑放 `world/block/` 或 `world/item/`
2. **注册中心集中** — 方块/物品/菜单/粒子等统一在 `init/` 注册
3. **数据生成器驱动** — 所有 JSON 由 datagen 生成，不手动编写
4. **RenderType 统一管理** — cutout / translucent 在 `ClientSetRenderLayer.java` 集中注册
5. **物品标签同步** — `ModItemTagsProvider` 用 `copy()` 从方块标签自动同步
6. **命名规范严格** — ID 语义化 snake_case，材质名与 ID 一致
7. **标签必补全** — 挖掘工具、材质类型、功能标签全覆盖
8. **世界生成走 datagen** — 维度/群系/地物/结构全部通过 `RegistrySetBuilder` + `DatapackBuiltinEntriesProvider` 产出 JSON
9. **升级配方保留 NBT** — 钻石→钛金等工作台升级配方使用 `NbtPreservingShapelessRecipe` / `NbtPreservingShapedRecipe`，`assemble()` 自动复制附魔
10. **技能系统链路** — 按键绑定（`ModKeyMappings`）→ 网络包（`BlinkPacket`）→ 技能核心（`BlinkSkill`）→ 效果处理（`EvasionEffectHandler`）→ 效果标记（`EvasionBuffEffect` + `BlinkCooldownEffect`）

## 五、辅助工具 `tools/`

项目根目录下 `tools/` 包含 ID 映射管理与 NBT 结构转换工具，详见 `tools/操作手册.md`。

| 工具 | 说明 |
|------|------|
| `generate_id_mapping.exe` | 从 `document/design/ID映射表.md` 生成 `mapping.json` |
| `lookup_id.exe` | 命令行查询旧 ID → 新 ID |
| `convert_structure.exe` | 将旧模组 `.nbt` 结构文件中的方块 ID 替换为新 ID |

### 映射数据流

```
ID映射表.md  →  generate_id_mapping.exe  →  mapping.json  →  convert_structure.exe
                                                    └─► lookup_id.exe
```

`mapping.json` 由 `generate_id_mapping.exe` 自动生成并纳入版本管理，位于 `document/design/mapping.json`。其中 `block_properties` 节为手动维护，重新生成时自动保留。
